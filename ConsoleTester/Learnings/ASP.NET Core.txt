[完了] LogLevel.None について書く

    通常、None == 0 とするが、LogLevel.None == 6 になっている。

    これは MS による enum のガイドラインに反するし、default で得られる (LogLevel) 0 が Trace なのは違和感がある。

    しかし、「○○以上をログ」というふるい分けなので、実装としては理由がある。「None 以上をログ」で6になると、Critical の5でも (5 >= 6) == false になって出力されない。

[完了] 学習メモをスタック方式と表現したことについて書く

    コミット前のメモのチェック時に、学習メモがスタック方式だという内容があった。

    もしかしてキュー方式でないかと思ったが、これは意外と興味深い問いだ。

    スタックなら、たとえばページを読んでいて、途中で分からないことがあったり、他のページへのリンクがあったりすれば、まずそちらを処理する。それが終わってから、途中まで読んだページに戻る。際限なく「先にこれを」へ移動するため、A → B → C → D → E くらいまで「先にこれを」が続いて、やっと終わって A まで戻ったとき、頭の中が浦島太郎ということがある。

    キューなら、途中で分からないことや他のページがあれば、タスクリストに入れて忘れる。浦島太郎にはならない。しかし、A の理解に B や C が有用でも A を進めることは、学びの質が高いとは言いがたい。

    もちろん現実的なのは両者の良いとこ取りで、「これはすぐに読んだ方が」というものもあるし、「これはあとでも」というものもあるので、個別に判断していくことになる。

    しかし、仮にどちらか一つの方式を徹底する場合、統計的にはどちらが学びの費用対効果に優れるのだろうか。

    たぶんスタック方式だろうと最初は思ったが、やっているとキュー方式も捨てがたい。

    というのも、ドキュメントは、たいていページ内にも内容の重複や説明の冗長性があり、A の前半ですぐに分からず、その時点では B や C を知る必要のあることでも、読み進めれば A の後半で何となく分かることもあるため。その場合、B や C を読むときに多くを飛ばせ、すぐに読み終わる。

    かたや、B や C は、A の理解の補助のみを目的に構成されているわけでなく、A に不要な情報も伴う。それが学びの多角化につながって、より知識が深まるのか、A を学んでいるコンテキストが頭の中で弱まることで A の理解度は少し下がるのか。

    ということを考えると、方式については、きちんと考えないわけにはいかない。そのため、このメモを書いている。

    暫定的な結論は、次のようになる。

    * A の学びのコンテキストが弱まる前に読み終わる程度の規模の B なら、まず B に移る（スタック方式）
    * 弱まりそうなら、B をタスク化して A を続行（キュー方式）
