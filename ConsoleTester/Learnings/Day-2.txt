Configuration in ASP.NET Core | Microsoft Learn を読み終えた
昨日、1/6ほどを1時間40分を掛けて読み、今日、残り5/6を1時間10分ほどで読んだ
ページ単位どころか、セクション単位でも内容が分かるくらい、コンテキスト情報がきちんと説明されるドキュメント
読めば読むほど、既に知っていて飛ばせることが多くなり、読むのが速くなってくる

おそらく10年くらい前だが、Web Forms から MVC に移行しなければならないと思い、少しかじったことがある
しかし、ちょっとしたフォームを作るだけでも、Entity Framework のルールに従ってモデルを作り、
    ビューを作り、コントローラーにコードを分離させて……という大げさなアプローチが、どうしても好きになれなかった
チームで開発するなら、そういう構造による役割分担が各部で「○○さん待ち」を減らすことのメリットは大きいだろう
一方、今なら no-code でも済むくらい簡単なものを一人で開発するにおいて、
    ごくわずかなコードのみ含むファイルを多数作るのは、ただただめんどくさかった

その頃にプログラミングから離れたのは、本業が忙しかったこともそうだが、MVC を好きになれず、先へ進めなかったことも大きい
JavaScript のフロントエンドのフレームワークと親和性が高かったのは MVC の方で、それで MVC が一気に広まった
自分は、MVC を好きになれなかったし、JavaScript でページをゴリゴリ動かすことも（少なくとも10年前には）業務システムにさほど必要でないと認識した
一方、Web Forms には、どんどんオワコンの空気が出てきていた
伸びている MVC + JavaScript に熱心になれず、オワコンの Web Forms にしがみつくのも将来性が乏しく、
    自分はどこを目指すべきか分からなくなったことでプログラミングに飽きては、プログラミング以外の仕事に軸足を移した

といった自己分析を書いているのは、自分が何に詰まったのか知ることが今後の学習につながるため

MVC をうるさく感じたのは自分だけでなかったようで、後続の Razor Pages が全面的に優れるとするページがある
Microsoft のページにも、Razor Pages can make coding page-focused scenarios easier and more productive than using controllers and views とある

これほど長くなると思わなかったのでテストコードのソースファイルにザッと書いてきたが、
    Razor Pages や JavaScript のフレームワークまで学ぶと、間違いなく数千行になる

VSC によるステージ・コミットのワークフローを保ちたい
Docs レポジトリーに入れて分離させることにはやはり懸念があるため、
    Testers ディレクトリーの隣に Learnings ディレクトリーを作るか

二つのレポジトリーを使い、片方に学んだこと、もう片方にそれに関連するコードを入れていくと、コミットが分散する
「こういうことを学んだから、こういうコードを書いたのか」というのが一つのコミットにまとまってほしい
いろいろと詰め込まれているのは、Testers ディレクトリーも同じ
Docs に入れれば Nekote のソリューションに学習メモがゴチャゴチャと入らないが、
    コミットが分散するのを避けたいため、Learnings でやってみる

ということで、メモを移動した
ファイル名には日付を入れず、Day-X[-Y].txt とした
X は、1からの連番
Y は、同じ日に2回以上学んだときのみ入る2からの連番
VSC は Nekote でいう alphanumeric sorting が可能だが、VS はそうでない
Day-10.txt 以降は「ソリューションエクスプローラー」でファイルの順序が不正確になるが、
    日ごとの管理において、その日のファイルを作れば VSC の Changes に入るため、特に問題なし

コメントに to-do をハイフンなしで書くと、VS は「タスク一覧」にそれらを表示する
iLoggingTester.cs に学習メモを書いていた頃はそれが便利だったが、テキストファイルでは検出されないようだ
Day-*.cs として全てをコメント行にするのはスマートでない
これだけのことで拡張機能を入れるのも避けたい
今後も to-do をハイフンなしで書き、学習時には「複数のファイル内を検索」の機能で探す
この作業の追加コストは軽微
