# Glossary of Generative AI Concepts in Nekote.Core DTOs

This document provides a detailed explanation of the various concepts, parameters, and fields present in the AI-related Data Transfer Objects (DTOs) within the `Nekote.Core` library. It is intended for developers who may be familiar with general IT concepts but new to the specifics of Large Language Model (LLM) and Generative AI APIs.

---

## 1. Core Generation Parameters

These parameters directly control the content and style of the text generated by the model.

### Temperature
- **Fields:** `generationConfig.temperature` (Gemini), `temperature` (OpenAI)
- **Concept:** Temperature controls the **randomness** of the model's output. At its core, a model predicts the next most likely word (or "token") from a probability distribution.
  - A **low temperature** (e.g., `0.1`) makes the model more deterministic and conservative. It will almost always pick the token with the highest probability. The output will be very predictable, factual, and less creative, which is good for tasks like summarization or question-answering based on a given context.
  - A **high temperature** (e.g., `1.0` or higher) increases randomness. It makes the model more willing to choose less likely tokens, leading to more creative, diverse, and sometimes surprising or nonsensical output. This is good for brainstorming, story writing, or creative tasks.
- **Analogy:** Think of it as a "creativity dial". Low is focused and high is experimental.

### Top P (Nucleus Sampling)
- **Fields:** `generationConfig.topP` (Gemini), `top_p` (OpenAI)
- **Concept:** Top P, or Nucleus Sampling, is another method to control randomness. Instead of considering all possible tokens, it considers only the smallest set of tokens whose cumulative probability exceeds the `top_p` value.
  - If `top_p` is `0.9`, the model considers the most likely tokens that add up to 90% probability and then chooses from that "nucleus" of tokens (factoring in `temperature`).
  - This is often considered a more robust way to control output than `temperature` alone, as it adapts the number of considered tokens based on the context. If the model is very confident about the next token, the nucleus will be small. If it's uncertain, the nucleus will be larger, allowing for more diversity.
- **Rule of Thumb:** It is generally recommended to alter either `temperature` or `top_p`, but not both at the same time.

### Top K
- **Fields:** `generationConfig.topK` (Gemini)
- **Concept:** Top K is the simplest method for controlling randomness. It instructs the model to only consider the `K` most probable tokens for the next choice.
  - If `top_k` is `40`, the model will limit its choice to the 40 most likely tokens at each step.
  - This can be less effective than Top P because it doesn't adapt. In some situations, the 40 most likely tokens might all be very low probability, while in others, the top 2 tokens might have 99% of the probability mass.

### Stop Sequences
- **Fields:** `generationConfig.stopSequences` (Gemini), `stop` (OpenAI)
- **Concept:** This allows you to define a list of custom strings that, if generated by the model, will immediately cause it to stop generating more text. This is useful for preventing the model from rambling or for structuring its output (e.g., you can tell it to stop when it generates the `

` sequence).

### Presence & Frequency Penalty
- **Fields:** `presence_penalty`, `frequency_penalty` (OpenAI)
- **Concept:** These are used to control the model's tendency to repeat itself.
  - **`frequency_penalty`:** A positive value penalizes tokens based on how frequently they have already appeared in the text, making it less likely for the model to repeat the exact same words or phrases.
  - **`presence_penalty`:** A positive value penalizes tokens simply for having appeared in the text at all, encouraging the model to introduce new topics and concepts.

### Seed
- **Fields:** `seed` (OpenAI)
- **Concept:** When you provide a `seed` (an integer), the model's output becomes deterministic (or at least, highly reproducible). If you run the same prompt with the same parameters and the same `seed`, you should get the exact same output. This is incredibly useful for testing, debugging, and situations where you need repeatable results. Without a seed, the output will have some level of randomness each time.

## 2. Logprobs (Logarithmic Probabilities)

- **Fields:** `logprobs`, `top_logprobs` (OpenAI), `avgLogprobs`, `logprobsResult` (Gemini)
- **Concept:** This is perhaps the most "new" concept. At each step of generation, the model doesn't just pick one word; it calculates a probability for *every possible* next token in its vocabulary.
  - **Probability:** The direct probability of a token, a number between 0 and 1.
  - **Log Probability (`logprob`):** For numerical stability and easier computation, models work with the natural logarithm of the probability. Since probabilities are between 0 and 1, their logs are always negative (or zero). A `logprob` closer to 0 indicates a higher probability.
- **Why is it useful?**
  1.  **Confidence Scoring:** You can see how "confident" the model was in the token it chose. If the chosen token has a very high probability (logprob close to 0), the model was very sure.
  2.  **Debugging:** You can inspect the `logprobs` to see what other tokens the model was considering, which can help you understand why it produced a certain output.
  3.  **Alternative Options:** The `top_logprobs` feature shows you the probabilities for the top N most likely tokens at each position, not just the one that was chosen. This allows you to see if there was a close "second choice" and potentially build features like auto-complete or alternative phrasing.

## 3. Tool Use & Function Calling

- **Fields:** `tools`, `tool_choice` (OpenAI, Gemini), `function_call` (OpenAI, deprecated)
- **Concept:** This is the mechanism that allows an LLM to interact with external systems. Instead of just generating text, you can give the model a list of "tools" (functions) it can use.
  - **`tools`:** A list of functions you define in your code, including their name, description, and parameters (using a format like JSON Schema).
  - **`tool_calls`:** When the model decides it needs to use one of your tools to answer a query, it stops generating text and instead returns a `tool_calls` object. This object contains the name of the function it wants to call and the arguments it wants to pass.
  - **Your Role:** Your code then receives this `tool_calls` object, executes your actual function with the provided arguments, gets a result, and sends that result back to the model in a subsequent API call.
  - **Final Answer:** The model then uses the result from your function to generate a final, human-readable answer.
- **Example:** You ask, "What's the weather in Tokyo?" The model, seeing it has a `get_weather(city)` tool, returns a `tool_calls` object asking to call that function with `city: "Tokyo"`. Your code runs the function, gets the weather data, sends it back, and the model then generates the text: "The weather in Tokyo is sunny with a high of 25°C."

## 4. Grounding, Retrieval & Citations

- **Fields:** `groundingAttributions`, `citationMetadata` (Gemini), `retrieval` (Context for a query)
- **Concept:** These concepts are central to Retrieval-Augmented Generation (RAG) and reducing hallucinations. They connect the model's generated statements back to specific source documents.
  - **Retrieval:** This refers to the process of finding relevant information from an external knowledge base (like a vector store) before generating an answer. The `Nekote.Core` RAG architecture design is based on this.
  - **Grounding:** This is the process of ensuring the model's output is based on ("grounded in") the retrieved information.
  - **Citations / Attributions:** These are the specific references returned by the model that link a part of its generated text to the exact source document or chunk it came from. The `citationMetadata` DTOs contain this information, allowing you to display footnotes or links next to the generated answer, proving where the information came from and allowing users to verify it.

## 5. Safety & Moderation

- **Fields:** `safetySettings`, `safetyRatings` (Gemini), `content_filter` (Finish Reason)
- **Concept:** Modern generative models have built-in classifiers to detect and block harmful content.
  - **`safetySettings`:** This is a parameter you send in your request, allowing you to configure the blocking threshold for different categories of harmful content (e.g., `HARM_CATEGORY_HARASSMENT`, `HARM_CATEGORY_HATE_SPEECH`). You can choose to be more or less strict.
  - **`safetyRatings`:** This is part of the response. The model returns its own assessment of both the user's prompt and its generated response for each category, along with the probability that it falls into that category.
  - **`content_filter` (Finish Reason):** If the model's output is blocked because it violated a safety setting, the `finish_reason` will often be `content_filter`, indicating that generation was stopped for safety reasons.
